## Problem Statement

目标站点已经证明了一件事：义务教育阶段的信息科技课程平台，不只是一个课程展示网站，而是一个高频、强流程、强课堂场景的教学业务系统。真实使用中，它至少承担了这些核心任务：

- 学生登录后查看当前学案、进入活动、提交作品、查看评价、做测验、练习打字、修改个人资料。
- 教师登录后按年级班级上课、发布课程与活动、查看签到、批改作品、配置量规、管理学生、维护题库和资源库。
- 整个平台围绕“上课 -> 活动 -> 作品提交 -> 教师评价 -> 数据统计”形成闭环。

但旧站的短板也很明显：

- 视觉风格老旧，信息密度高，学生端和教师端的认知负担都偏大。
- 明显偏桌面机房时代，缺少面向学校综合展示的门户层。
- 功能入口多但组织方式不稳，很多能力分散在独立页面中。
- 系统健壮性不足，例如教师“状态”页面在 2026-03-24 实测访问时出现了 `NullReferenceException`。

新平台不该只是把旧页面换成 Vue，而应该升级成一个集“学校门户 + 课程学习 + 教学管理 + AI 助教 + 数据分析”于一体的现代化综合平台。

## Status Quo

### 学生端现状

- 核心导航包含：`我的学案`、`我的作品`、`打字宝典`、`常识积累`、`在线资源`、`我的网盘`
- 课程结构为：`学案 -> 活动`
- 活动页支持：
  - 阅读任务说明
  - 下载材料
  - 上传作品
  - 查看是否已提交
  - 发起作品互评
- 个人资料支持：
  - 小组申请
  - 密码修改
  - 班级修改
  - 相片 / 性别 / 姓名修改

### 教师端现状

- 一级导航包含：`上课`、`备课`、`量规`、`作品`、`签到`、`学生`、`常识`、`中文`、`拼音`、`资源`、`信息`、`状态`、`帮助`
- “备课”支持：
  - 按年级和学期管理学案
  - 发布 / 编辑 / 打包 / 分析课程
  - 在课程下添加活动、讨论、调查、填表
  - 添加不同类型的编程活动：Scratch、Python、流程图、像素画、HTML
- “量规”支持按作品类型维护评价标准
- “作品”支持查看未评分作品、按活动查看、按等级评分
- “签到”支持查看每日签到总表和明细
- “学生”支持名单、密码、作品、表现、小组与权限控制
- “常识”是题库管理
- “资源”是分类资源库

### 当前 AI 机会点

旧站其实已经暴露出几个最适合 AI 增强的切口：

- 教师发布活动时，常常需要重复写任务说明、材料说明、题目和评分标准。
- 学生完成的作业越来越多样，不再只是文件上传，也可能是交互答题、图像作品、代码、截图、文档。
- 教师端已有大量作品和签到数据，但目前分析维度有限，缺少自动汇总和讲评能力。
- 旧站活动类型已经很多，说明“活动引擎”本身就是平台核心，AI 最适合在这里产生复利。

## Target User & Narrowest Wedge

### Target User

最具体的核心用户不是“学校”或“教师群体”，而是下面两类角色：

- 一位义务教育阶段的信息科技教师，需要在一节课内完成课程发布、学生签到、作业发放、数据回收和讲评。
- 一位七到九年级学生，需要快速进入当天课程、完成活动任务、提交结果并查看反馈。

### Narrowest Wedge

最小可上线且真正有价值的版本，不是全量复刻旧站所有辅助能力，而是：

1. 一个公共门户首页
2. 一个学生学习中心
3. 一个教师课堂工作台
4. 一条完整的“课程 -> 活动 -> 作业 -> 提交 -> 评价 -> 分析”链路
5. 一个可用的 AI 作业工坊
6. 一个课程专属智能体能力
7. 一套基础班级与签到管理

只要这一条主链路做顺，新平台就已经具备替代旧站核心业务的能力，并且开始形成旧站没有的新价值。

## Constraints

- 场景是义务教育阶段，页面语言和交互必须足够简单，避免企业化过度复杂。
- 教师端需要兼容机房课堂场景，优先保证桌面端效率；学生端需要兼顾平板和手机。
- 必须支持多班级、多学期、多课节、多活动。
- 必须支持多种作品类型和文件预览，包括图片、Office 文档、代码文件、压缩包等。
- 旧站积累了大量课程、作品、学生与签到数据，新平台设计时要预留迁移接口。
- 课程内容和资源涉及校内教学资产，需要考虑校内服务器部署和权限隔离。
- AI 生成功能必须可控，不能让教师直接发布未经审核的高风险内容。
- 学生端智能体必须默认是“启发式辅导”，不能直接沦为标准答案输出器。
- 数据库前期应尽量轻量，便于快速试点，但后期必须可迁移到更强的生产架构。
- 一期即使先服务单校，也必须从数据模型、权限模型、文件存储和 AI 配额上按多校租户边界设计。
- 旧站的机房课堂能力需要保留，至少覆盖机房视图、IP 锁定登录、班级密码和课堂状态控制。

## Premises

以下前提是本次设计的工作假设：

1. 新平台的核心不是“宣传网站”，而是“教学业务系统 + 学校展示门户 + AI 教学能力”的组合。
2. 课程的原子结构应为 `课程 / 学案 -> 活动 -> 作答过程 -> 提交记录 -> 评价记录 -> 分析结果`。
3. 教师端必须保留旧站“一屏控课堂”的优势，但要用更现代的工作台呈现方式重构。
4. 学生端首页必须围绕“今天做什么”设计，而不是围绕“系统有什么模块”设计。
5. 教师发布 AI 作业时，不建议让模型直接输出任意 HTML 后立即上线，而应先生成结构化活动描述，再由前端渲染。
6. 课程专属智能体必须是“按课程隔离、按权限隔离、按知识库隔离”的，而不是全局无限制共享。
7. 一期优先做 Web 平台，原生 App 不是当前必要条件。

## Approaches Considered

### Approach A: 旧站功能平移版

Summary:
把旧站按原有菜单结构迁移到 `Vue + FastAPI`，页面名称和业务分组保持尽量一致，快速完成技术升级。

Effort: M
Risk: High
Pros:

- 功能映射最直接，方便旧用户迁移
- 开发决策简单，需求争议较少
- 对数据迁移和权限模型改动较小

Cons:

- 会把旧站的信息架构问题一起复制过来
- 无法真正体现“综合网站”应有的门户能力和学校形象
- AI 能力只能零散外挂，无法形成平台核心能力

Reuses:

- 旧站菜单与业务模块命名
- 旧站课程、活动、作品、签到的基本流程

### Approach B: 综合门户 + 双工作台版

Summary:
建立一个统一的学校信息科技平台，拆成“公共门户、学生中心、教师工作台、管理后台”四层，保留旧站主链路并重构入口与界面。

Effort: L
Risk: Medium
Pros:

- 既保留教学闭环，又能承担学校综合网站形象
- 学生端和教师端都能围绕高频任务重构
- 架构更适合后续扩展通知、课程地图和数据分析

Cons:

- AI 能力仍然只是附加模块，无法最大化发挥差异化优势
- 活动生成和智能体能力没有天然落位

Reuses:

- 旧站真实业务流程
- 旧站中的课程树、活动模型、签到和作品评价机制

### Approach C: 综合门户 + 双工作台 + AI 活动引擎版

Summary:
在现代化门户和双工作台基础上，引入“AI 作业工坊 + 课程专属智能体 + 多模态分析中台”，把 AI 从外挂能力升级为平台核心引擎。

Effort: L-XL
Risk: Medium
Pros:

- 最贴合你这次新增的 AI 作业、课程智能体和多模态诉求
- 让教师发布课程、布置作业、分析结果的效率显著提升
- 形成平台的长期壁垒，而不是只做 UI 重构

Cons:

- 需要额外设计 AI 治理、成本控制、审计和安全边界
- 一期实现范围必须克制，否则容易过大

Reuses:

- 旧站课程 / 活动 / 作品主链路
- 旧站教师工作台的课堂组织方式

**RECOMMENDATION:** 选择 `Approach C`，因为它既能保留旧站已验证的教学闭环，又能把你明确要求的 AI 能力做成平台级能力，而不是零散插件。

## Recommended Approach

推荐做成一个四层结构的平台，并新增一个 AI 能力层：

### 1. 公共门户

面向未登录用户、家长、校方领导和对外访问者：

- 首页
- 学校 / 学科介绍
- 课程地图
- 通知公告
- 优秀作品展示
- 教研成果 / 资源推荐
- 统一登录入口

### 2. 学生中心

围绕“今日任务”和“成长记录”设计：

- 我的课程
- 本周任务
- 活动详情
- 交互作业页
- 作品提交与历史记录
- 测验练习
- 资源中心
- 个人资料 / 小组 / 签到记录

### 3. 教师工作台

围绕“备课、上课、收作业、看数据”设计：

- 课堂总控
- 课程管理
- AI 作业工坊
- 活动编辑器
- 作品批改中心
- 签到管理
- 学生管理
- 题库测验
- 资源库
- 数据分析

### 4. 管理后台

面向校级管理员和教研管理员：

- 用户与角色管理
- 年级班级管理
- 站点内容管理
- 文件存储策略
- AI 模型与额度配置
- 日志与审计

### 5. AI 能力层

新增一层 AI 中台能力，统一为门户、学生端和教师端服务：

- AI 作业生成
- 智能体运行时
- 多模态解析
- 讲评总结
- 题目生成
- 数据分析摘要
- Prompt 模板与版本管理

## AI Native Additions

### A. AI 作业工坊

教师发布作业时，可以输入：

- 年级
- 课程主题
- 学习目标
- 题型偏好
- 难度
- 评分规则
- 参考材料
- 希望的交互形式

系统调用 AI 接口后生成：

- 作业标题
- 作业说明
- 页面结构
- 题目与选项
- 图片 / 文本 / 文件引用
- 自动评分规则
- 教师讲评提示词

#### 关键推荐

不建议让模型直接输出任意 HTML 后上线。

更稳妥的做法是：

1. AI 先生成结构化 `ActivitySpec` 或 `AssignmentSchema`
2. Vue 前端用统一渲染器把它渲染成交互作业网页
3. 教师预览后才能发布
4. 如果确实要支持自由 HTML，则必须运行在沙箱 iframe 中

这样做的好处：

- 更安全
- 可控性更强
- 方便统计每一道题的作答数据
- 后续更容易做题型扩展和数据分析

#### 一期组件策略

- 一期允许少量受控自定义题型组件，但必须以白名单方式注册进统一渲染器。
- 每个组件都要定义明确的输入 schema、评分适配器和回传事件格式，不能让教师直接注入任意 JS/HTML。
- 推荐首批仅开放：拖拽排序、热点标注、流程连线、图片判读 4 类组件。

### B. 作答数据与成绩回流

AI 作业页不只是“可点击页面”，而是一个完整的数据采集单元。

系统需要记录：

- 学生开始时间
- 每题作答过程
- 每次修改记录
- 最终答案
- 自动评分结果
- 教师复核分数
- 用时
- 提交设备信息

教师后台可直接看到：

- 班级整体正确率
- 题目通过率
- 高错误率题目
- 平均用时
- 分数分布
- 优秀作答样例
- 需要重点讲评的知识点

### C. 课程专属智能体

教师发布课程时，可选择“创建本课程专属智能体”。

该能力作为教师端独立模块上线，同时在课程上下文中提供快捷入口，由教师自行决定某门课程是否添加、绑定、启用或停用。

该智能体可由 AI 辅助完成初始化，输入源包括：

- 课程标题
- 教学目标
- 教案
- PPT
- 题库
- 量规
- 参考网站
- 教师上传的图片、文档、视频或截图

课程智能体建议至少支持 4 种角色：

- `课程助教`
  - 帮教师生成活动、补充例题、设计讲评
- `学生答疑助手`
  - 为学生提供启发式提示和步骤引导
- `作业讲评助手`
  - 自动归纳全班共性问题和优秀案例
- `资源整理助手`
  - 把教师上传材料归类成可检索知识库

课程智能体允许接入外部网站作为知识源，但必须满足以下边界：

- 由教师显式添加和启用，不默认全网开放抓取
- 受学校或租户级域名白名单控制
- 抓取后保留来源 URL、抓取时间和内容快照，方便审计与复现
- 回答时优先引用课程资料与已入库内容，外部网站仅作补充知识源

### D. 侧边栏智能体助手

整个平台增加一个右侧可折叠的“智能体助手”。

#### 在教师端

- 读取当前页面上下文
- 支持快速提问：
  - 帮我生成本课作业
  - 根据这节课内容出 5 道基础题
  - 总结本班最常错的 3 个点
  - 生成课堂讲评提纲
  - 基于这组学生作品给出讲评

#### 在学生端

- 提供启发式辅导，不直接给标准答案
- 支持：
  - 我没看懂题目
  - 帮我分步骤理解
  - 这张图是什么意思
  - 这段代码哪里有问题

### E. 多模态能力落位

平台应明确利用大语言模型的多模态能力，而不是只做文本问答：

- 识图：
  - 识别学生上传的海报、流程图、界面截图、思维导图
- OCR：
  - 把教材页、拍照作业、手写草图转成结构化文本
- 文档理解：
  - 解析 Word、PPT、PDF，抽取知识点和题目
- 代码理解：
  - 分析 Python、HTML、Scratch 截图或代码文件
- 讲评生成：
  - 基于图片、文字、代码、成绩等多源信息生成课堂讲评

一期聚焦图片、OCR、文档和代码理解；视频多模态能力明确放到二期，不进入首批上线范围。

## Key User Flows

### 学生流程

1. 登录学生端
2. 进入“今日任务”卡片
3. 打开某课程下的活动或 AI 作业页
4. 阅读任务说明、在线作答或上传作品
5. 作答过程与成绩自动回传后台

### 教师流程

5. 让 AI 生成交互作业页草案

### 数据库演进建议

你提出“前期用 SQLite，成熟后迁移到 PostgreSQL”，这是合理的。

推荐落地方式：

1. 从第一天起就使用 ORM 和 Alembic
2. 避免写大量 SQLite 专属 SQL
3. 复杂 JSON 数据在 SQLite 阶段可先以 `TEXT / JSON 字符串` 存储
4. 业务量和并发变大后，迁移到 PostgreSQL 的 `JSONB`、索引和更强事务能力

推荐演进路线：

- `Phase 1`
  - FastAPI + SQLite + Alembic
  - 用于试点学校、单机部署、快速验证
- `Phase 2`
  - PostgreSQL 替换 SQLite
  - Redis 接入缓存、队列与会话
  - 适配多班级并发、分析任务和 AI 异步任务

### 多校租户边界

- 从一期开始引入 `tenant_id`，哪怕首个试点只有一所学校。
- 用户、班级、课程、资源、AI 额度、审计日志、对象存储路径都按租户隔离。
- 首个学校可以作为默认租户试点部署，但代码、数据库 schema 和权限判断不能写死成单校模式。

### 作业发布与作答主链路

```text

          |

          v
```

规则：只要学生已经开始作答，就不能原地改写已发布版本；后续修改必须生成新的 `ActivityRevision` 和 `ActivityPublication`。

## Additional Recommendations

除了你明确要求增加的方案，我推荐再补上这些：

### 1. AI 输出审核机制

- AI 生成的作业、讲评和课程助手设定，默认都需要教师确认后才能发布。

### 2. Prompt 与版本管理

- 每次生成作业或课程智能体时，记录所用 Prompt 模板和模型版本，方便复现和回滚。

### 3. 成本与额度控制

- 教师、课程、学校三个层级都要有 AI 调用额度统计。

### 4. 故障降级机制

- AI 服务异常时，教师仍可手动创建普通活动，不能让课堂流程被 AI 卡死。

### 5. 学生智能体防“直接给答案”

- 学生侧默认采用“分步提示、反问、纠错、引导”模式，而不是直接给标准答案。

### 6. 多模态作品讲评面板

- 支持教师上传图片、PDF、PPT、代码、截图后，让 AI 自动生成讲评草稿。

### 7. 课程知识库检索

- 课程智能体最好基于课程专属知识库回答，而不是纯模型自由发挥。
- 若允许外部网站接入，也必须先转成可审计的知识源快照，再进入课程知识库。

### 8. 审计与隐私保护

- 所有 AI 对话、生成和批量分析都要留审计日志。
- 学生隐私数据不能被跨课程、跨班级智能体共享。
